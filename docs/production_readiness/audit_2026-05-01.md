repo split. It focuses on package structure, API shape, native packaging,
documentation, test posture, and known architecture risks.

## Strengths

- The editor already has a substantial regression suite across editing,
  projection, markdown features, command APIs, native parity, and benchmarks.
- Runtime parsing is backed by native `comrak`, with explicit UTF-8/UTF-16
  offset mapping and parity fixtures.
- Read-only preview (`SovereignMarkdownView`) shares the editor parse/render
  pipeline, which reduces edit/preview drift.
- Existing architecture RFCs identify the right long-term direction:
  controller facade, composed session state, explicit pipeline/syntax/intent
  modules, and command-layer ownership.

## Findings

### P0: Native packaging is not production package quality yet

The native bridge builds through hand-authored scripts and still requires
consumer-app wiring for mobile artifacts. The standalone repo now stages mobile
outputs under `native/comrak_bridge/dist`, but there is no Dart/Flutter native
assets build hook, package template structure, or example app packaging harness.

Production target:

- use a first-class FFI package build model (`hook/build.dart`, generated
  bindings where appropriate, and documented artifact flow), or a deliberate
  plugin/package hybrid if Flutter plugin APIs become necessary;
- provide a repeatable example app/harness for Android and iOS verification;
- make desktop/mobile loading behavior deterministic for consumers.

### P0: Public API surface is too broad

`lib/sovereign_editor.dart` exports controller, widgets, engine internals,
low-level scanners, models, and parsing utilities. Because these files live
outside `lib/src`, many implementation details are also importable by package
consumers through deep imports.

Production target:

- define a small stable public API;
- move internals behind `lib/src`;
- keep only intentional public libraries under `lib/`;
- document migration notes for any breaking API cleanup.

### P1: Naming still carries Dune-specific vocabulary

The extracted package still exposes `DuneMarkdownTheme` naming. Initial
extraction also exposed `AppColors`; the first Phase 1 cleanup removed that
public helper and made the fallback palette private to the markdown theme. The
remaining Dune theme name is understandable history, but it makes the package
feel app-bound and will leak Dune vocabulary into consumer code.

Production target:

- rename the public markdown theme object to Sovereign vocabulary;
- preserve compatibility only if explicitly required and documented.

### P1: Controller and native bridge need further decomposition

`SovereignController` and the Rust bridge remain large orchestration-heavy
files. Existing RFC 017 already describes the desired module boundaries. The
current shape is testable but still has too much responsibility in single files.

Production target:

- continue the controller facade extraction;
- split intent routing, editing pipeline, structure query/transform, projection
  coordination, and rendering services into small typed modules;
- split the Rust bridge into parse mapping, ABI response construction, and
  conformance helpers.

### P1: Release metadata and package docs are incomplete

The repo has `README.md`, but no `LICENSE`, `CHANGELOG.md`, `example/`, release
guide, API docs policy, screenshots, repository metadata, or explicit platform
matrix in `pubspec.yaml`. `publish_to: none` remains correct until these are
resolved.

Production target:

- decide license before any public distribution;
- add changelog, example app, screenshots if useful, repository/issue tracker,
  supported platform declarations, and dry-run publishing checks;
- keep generated files out of source control.

### P2: Historical docs still reference Dune paths and rollout context

The copied architecture docs are valuable but still describe Dune integration
and old monorepo commands. Treat them as historical context until rewritten for
the standalone package.

Production target:

- split historical RFCs from current package docs;
- update current architecture docs to root-relative paths and package-neutral
  language;
- add a "consumer integration" guide for native artifacts.

### P1: Enforced benchmark lane required remediation

Initial extraction found that the regular test suite printed benchmark warnings
without failing, and the explicit benchmark gate failed on this machine when
budgets were enforced:

- warm `buildTextSpan` cache hit: 10896us vs 800us budget
- scanner p99: 8327us vs 3500us budget
- cold `buildTextSpan` p99: 12374us vs 4000us budget
- emoji scanner p99: 5825us vs 3000us budget

Follow-up status:

- resolved on 2026-05-01 by stabilizing default theme identity, adding a
  final `TextSpan` cache, avoiding redundant fenced-code scans in renderer
  cache misses, skipping link/image supplemental scanning when no link/image
  triggers exist, and making the benchmark harness measure warmed steady-state
  p99 with a real Flutter `BuildContext`;
- `./scripts/verify_benchmark_lane.sh` now passes with enforced budgets;
- `./scripts/verify_release.sh` now passes end to end.

Continuing production target:

- keep benchmark thresholds calibrated as hardware and renderer architecture
  change;
- reduce noisy benchmark output in the ordinary full test suite if it becomes
  distracting.

### P2: Verification still needs a complete release gate

The package has strong tests, but production readiness needs a clearly named
release gate that covers analysis, fast regressions, full test suite, native
host build, mobile packaging harnesses, benchmark budgets, docs generation, and
publish dry-run.

Production target:

- define `scripts/verify_release.sh`;
- make the confidence gate fast and the release gate comprehensive;
- record every gate result in `docs/production_readiness/execution_log.md`.
